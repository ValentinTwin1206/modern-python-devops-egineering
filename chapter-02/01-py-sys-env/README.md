# Python System Environment

This project explains what happens on a Linux system when Python is
installed and packages are installed into it. It includes a tiny Bottle
web server, a Karva test, Ruff linting, and a Dockerfile that inspects a
real Ubuntu-based Python installation.

The web server is intentionally small. The focus is understanding Python
on Linux: where the interpreter lives, where packages are installed, what
the standard library is, how imports are resolved, and why system Python
is different from project environments.

## 1. Introduction

Python on Linux is both an executable and a directory structure. The
executable runs your code. The directory structure contains the standard
library, third-party packages, command-line scripts, package metadata,
headers for native extensions, and sometimes virtual-environment
configuration.

This folder uses a tiny application to keep the discussion concrete:

- Bottle provides the HTTP server.
- Karva runs the test.
- Ruff checks the code.
- `uv` manages local project dependencies and performs system-level
  package installation in Docker.

The Dockerfile is the important system-level demonstration. It starts
from `ubuntu:26.04`, installs Python with APT, installs `uv`, then uses
`uv pip install --system --break-system-packages bottle karva ruff` to
install packages into the system interpreter instead of a virtual
environment. The extra flag is required because modern Ubuntu marks its
APT-managed Python as externally managed.

## 2. Where Python Lives On Linux

On Debian and Ubuntu, Python installed by the operating system package
manager is normally exposed as:

```text
/usr/bin/python3
```

That interpreter is managed by APT. Its files are owned by Debian
packages such as `python3`, `python3-minimal`, and `python3-pip`.

Another common path is:

```text
/usr/local/bin/python
```

That path is often used for manually installed Python builds and for
container images that ship Python outside the distribution package
manager. The official `python:3.11-slim` image usually puts Python under
`/usr/local`. The Ubuntu image used here installs Python under `/usr`
through APT.

Ask the shell which executable it will run:

```sh
which python
which python3
```

On Ubuntu, `python3` is normally available. The bare `python` command may
not exist unless an administrator or package creates that alias.

## 3. Python Installation Folder Structure

A Python installation commonly uses this conceptual layout:

```text
bin/
include/
lib/pythonX.Y/
lib/pythonX.Y/site-packages/
lib64/
pyvenv.cfg
```

Not every directory appears in every installation. A system interpreter,
a manually installed interpreter, and a virtual environment share the
same ideas but use different concrete paths.

### `bin/`

`bin/` contains executable commands. In a Python installation, this can
include:

- `python`
- `python3`
- `pip`
- `pip3`
- console scripts installed by packages, such as `ruff`

Some entries are real executables and some are symlinks. For example,
`python3` may point to a versioned executable such as `python3.13`, and
`pip3` may point to a versioned pip command.

Shell execution is controlled by `PATH`. If one directory appears before
another in `PATH`, commands in the earlier directory win. Virtual
environment activation uses this rule by putting the environment's
`bin/` directory at the front of `PATH`.

Inspect it with:

```sh
echo $PATH
which python3
which ruff
```

### `include/`

`include/` contains C header files for Python. These headers are needed
when a package builds a native extension module, such as a `.so` file
implemented in C, C++, or Rust.

Pure Python packages usually do not need these headers. Packages with
compiled components need them because they compile against Python's C
API.

### `lib/pythonX.Y/`

`lib/pythonX.Y/` contains the standard library for a specific Python
feature version. For Python 3.13, the directory is named `python3.13`.
For Python 3.11, it is named `python3.11`.

The standard library is the set of modules shipped with Python itself.
Examples include:

```python
import os
import sys
import json
import pathlib
```

Those imports work without installing packages from PyPI because they are
part of the Python installation.

### `site-packages/`

`site-packages/` is where Python packaging tools install third-party
packages. Bottle, Karva, and Ruff are not part of the standard library,
so they must be installed before you can import or run them.

For a manually installed Python, packages often go under a path like:

```text
/usr/local/lib/pythonX.Y/site-packages/
```

For a virtual environment, packages go under:

```text
.venv/lib/pythonX.Y/site-packages/
```

### `dist-packages/`

Debian and Ubuntu also use `dist-packages`. You may see paths like:

```text
/usr/lib/python3/dist-packages/
/usr/local/lib/python3.X/dist-packages/
```

This is Debian policy. It helps distinguish distribution-managed Python
packages from packages installed by Python-specific tooling. The exact
path depends on the interpreter, distribution, and installation scheme.

### `lib64/`

On some 64-bit Linux distributions, `lib64/` exists for architecture
compatibility. It may be a real directory or a symlink to `lib/`. Python
and packaging tools can use it when the platform separates 64-bit
libraries from other library files.

Ubuntu containers often do not emphasize `lib64/`, but you will see it
on some distributions and virtual environments.

### `{name}-{version}.dist-info/`

Installed Python packages usually include metadata directories such as:

```text
bottle-0.13.4.dist-info/
ruff-0.15.12.dist-info/
```

These directories contain metadata used by installers and tooling:

- package name and version
- dependencies
- installed file records
- entry points such as console scripts
- wheel metadata

Tools such as `uv`, `pip`, `importlib.metadata`, and build frontends use
this metadata to understand what is installed.

### `pyvenv.cfg`

`pyvenv.cfg` appears at the root of a virtual environment. It records how
the environment relates to its base Python installation. A typical file
contains values like:

```ini
home = /usr/bin
include-system-site-packages = false
version = 3.13.0
```

This file helps Python decide that it is running inside a virtual
environment. It affects prefix values and import path setup, including
whether the environment can see packages from the system interpreter.

## 4. Standard Library Vs Installed Packages

The standard library ships with Python:

```python
import os
import sys
```

Third-party packages must be installed:

```python
import bottle
import karva
```

If `import os` works, that proves Python itself is installed. If
`import bottle` works, that proves Bottle is present somewhere on
Python's import path, usually in `site-packages` or `dist-packages`.

## 5. PATH And Execution

When you type a command, the shell searches `PATH` from left to right.
It does not search the whole filesystem.

```sh
echo $PATH
which python3
which ruff
```

If `/usr/local/bin` appears before `/usr/bin`, a manually installed
Python in `/usr/local/bin/python` may win over an APT-managed Python in
`/usr/bin/python3`. If a virtual environment is active, `.venv/bin`
usually appears first.

Python imports have their own search path, `sys.path`:

```sh
python3 -c "import sys; print(sys.path)"
```

Executables are resolved by the shell through `PATH`. Imports are
resolved by Python through `sys.path`.

## 6. System Python Vs Project Environments

System Python belongs to the operating system. On Debian and Ubuntu, the
system interpreter and many packages live under paths like:

```text
/usr/bin/python3
/usr/lib/python3/dist-packages/
```

A project virtual environment keeps dependencies separate:

```text
.venv/bin/python
.venv/lib/pythonX.Y/site-packages/
```

The isolation matters because global installs can collide. If one
project installs a newer dependency into the system interpreter, another
project or system tool may unexpectedly import that newer version.

For day-to-day development, prefer project environments. This README
shows system-level installs because they are useful for understanding how
Linux Python is structured, not because they are the safest default on a
real workstation.

## 7. System-Level Package Installation With `uv`

`uv` can install packages into the active project environment, a virtual
environment, or the system interpreter. For system-level installation,
the core form is:

```sh
uv pip install --system bottle karva ruff
```

That tells `uv` to target the system Python installation instead of
creating or using a project virtual environment.

On modern Debian and Ubuntu, the system interpreter is usually marked as
externally managed by PEP 668. In that case, Python packaging tools
refuse to write into the system interpreter unless you explicitly accept
the risk. The Dockerfile demonstrates the working container pattern:

```sh
uv pip install --system --break-system-packages bottle karva ruff
```

Use that override only in disposable environments such as containers. On
a real workstation or shared server, prefer a virtual environment.

You may also see:

```sh
uv pip install bottle karva ruff
```

That form installs into the environment selected by `uv`. Inside a
virtual environment, it installs into that virtual environment. With
`--system`, the target is explicit.

After a system install, packages end up in the system interpreter's
package directory. On Ubuntu, that may be a `dist-packages` or
`site-packages` path depending on the interpreter, install scheme, and
tooling. Inspect the actual location with:

```sh
python3 -m site
python3 -c "import bottle; print(bottle.__file__)"
python3 -c "import sysconfig; print(sysconfig.get_path('purelib'))"
```

This is useful in containers, where the image is disposable and owned by
the application. It is risky on real systems, where the same Python
installation may be used by OS tools and unrelated applications.

## 8. Insights From The Ubuntu 26.04 Docker Image

The Dockerfile uses:

```dockerfile
FROM ubuntu:26.04
```

It installs Python through APT:

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        python3 \
        python3-pip
```

That means Python is the Ubuntu system interpreter, not a Python copied
into `/usr/local` by the official Python image.

The Dockerfile then installs packages into that system interpreter. Since
Ubuntu marks the interpreter as externally managed, the container uses an
explicit override:

```dockerfile
RUN uv pip install --system --break-system-packages bottle karva ruff
```

During the build, it prints:

```sh
which python3
python3 --version
python3 -m site
python3 -c "import sys; [print(path) for path in sys.path]"
python3 -c "import sysconfig; print(sysconfig.get_path('purelib'))"
python3 -c "import bottle, karva, ruff; print(bottle.__file__); print(karva.__file__); print(ruff.__file__)"
```

Those commands answer the practical questions:

- where the interpreter is located
- which Python version APT installed
- where Python searches for imports
- where system-level packages were installed by `uv`
- where Bottle, Karva, and Ruff physically live on disk

This differs from `python:3.11-slim`. The official Python image ships a
Python installation under `/usr/local`, while the Ubuntu image installs
Python as an operating system package under `/usr`. Both are Linux, but
their ownership model and default paths differ.

## 9. Project Example: Tiny Webserver

The application is intentionally minimal:

```python
from bottle import Bottle, response, run

app = Bottle()


@app.route("/")
def index():
    response.content_type = "application/json"
    return {"message": "Hello from tiny webserver"}
```

Run it locally as a normal `uv` project:

```sh
uv sync
uv run tiny-webserver
```

Run the development checks:

```sh
uv run karva test tests/
uv run ruff check .
```

Inside the Docker image, the app runs with system-installed Bottle and
`PYTHONPATH=/app/src`:

```sh
docker build -t py-sys-environment .
docker run --rm -p 8080:8080 py-sys-environment
```

Then request:

```sh
curl http://localhost:8080/
```

Expected response:

```json
{"message": "Hello from tiny webserver"}
```

## 10. Installing Dependencies Without A Virtual Environment

On Debian or Ubuntu, install Python and pip with APT:

```sh
sudo apt update
sudo apt install python3 python3-pip
```

Then install packages globally with pip:

```sh
pip3 install bottle karva ruff
```

Or install them globally with `uv`:

```sh
uv pip install --system bottle karva ruff
```

On current Debian/Ubuntu releases, that command may fail with an
"externally managed" error. The forceful equivalent is:

```sh
uv pip install --system --break-system-packages bottle karva ruff
```

That flag is intentionally loud because it can overwrite or shadow files
expected by the operating system package manager.

Both approaches install into directories used by the system Python
installation, such as `dist-packages` or `site-packages`. The exact path
depends on the distribution and install scheme.

The tradeoff is important:

- System installs are simple in containers and disposable environments.
- System installs are risky on developer machines and servers shared by
  multiple applications.
- Virtual environments avoid most of that risk by giving each project
  its own interpreter entry points and package directory.

The key lesson is that installing a package means placing importable
files plus metadata somewhere on Python's import path. Whether that path
is global or project-local determines who can see those packages and who
can be affected by version changes.
